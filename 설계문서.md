## 0. 설계 전 확정 사항 (가정값)

SRS에서 미결정으로 남겨둔 항목을 설계 진행을 위해 다음과 같이 **가정**한다. 변경이 필요하면 이 절만 수정하고 관련 설계를 따라 조정한다.

| 항목 | SRS 참조 | 가정한 값 | 이유 |
|---|---|---|---|
| 슬롯당 예약 인원 | BR-02 | **4명** (capacity = 4) | 한 시간대에 최대 4명까지 동시 수업 수용. 슬롯 생성 시 사장님이 변경 가능. |
| 예약 마감 기준 | BR-01 | 예약일 **전날 23:59:59**까지 요청·수락 가능, 그 이후 마감 | "예약 당일 기준 1일 전" 을 가장 직관적으로 해석 |
| 알림 방식 | 2.5 | **앱 내 알림(In-App Notification)** 으로 1차 구현 | 외부 채널(문자/카톡) 연동은 비용·인증 이슈로 확장 사항 |

## 1. 시스템 아키텍처

### 1.1 전체 구성

```
┌─────────────────┐      ┌─────────────────┐      ┌──────────────┐
│  회원 브라우저   │      │ 사장님 브라우저  │      │              │
│  (예약 화면)     │      │  (관리 화면)     │      │              │
└────────┬────────┘      └────────┬────────┘      │              │
         │   HTTP / REST API      │               │              │
         └───────────┬────────────┘               │              │
                     ▼                            │              │
         ┌───────────────────────────┐            │              │
         │   Spring Boot 애플리케이션 │            │              │
         │  ┌─────────────────────┐  │            │              │
         │  │ Controller (REST)   │  │            │              │
         │  ├─────────────────────┤  │            │              │
         │  │ Service (비즈니스)  │  │ ◀────────▶ │   Database   │
         │  ├─────────────────────┤  │   JPA      │ (MySQL 기준) │
         │  │ Repository (JPA)    │  │            │              │
         │  └─────────────────────┘  │            │              │
         └───────────────────────────┘            └──────────────┘
```

### 1.2 아키텍처 스타일

- **계층형 아키텍처(Layered Architecture)**: Controller → Service → Repository → DB
- 화면(View)과 API를 분리하여, 추후 모바일 앱이 같은 REST API를 재사용할 수 있도록 한다 (NFR-07).
- 1차 웹 화면은 서버 사이드 템플릿(Thymeleaf) 또는 간단한 정적 페이지 + fetch 중 택1. **본 설계는 REST API + 별도 화면**을 기준으로 한다.

### 1.3 기술 스택

| 구분 | 선택 | 비고 |
|---|---|---|
| 언어 | Java 17 | LTS 버전 |
| 프레임워크 | Spring Boot 3.x | Spring Web, Spring Data JPA, Spring Security |
| ORM | JPA (Hibernate) | |
| DB | MySQL 8.x | 본 문서는 MySQL 기준. PostgreSQL 사용 시 JPA 덕분에 코드 변경 거의 없음 (DDL 타입만 일부 차이) |
| 빌드 | Gradle | |
| 인증 | 세션 기반 로그인 (Spring Security) | 학습 난이도 고려. JWT는 확장. |
| 화면 | Thymeleaf 또는 HTML+JS | 40~50대 대상, 단순 화면 우선 |

## 2. 데이터베이스 설계

### 2.1 ERD (개념도)

```
┌──────────────┐         ┌──────────────────┐         ┌──────────────────┐
│    users     │         │    time_slots    │         │   reservations   │
├──────────────┤         ├──────────────────┤         ├──────────────────┤
│ id (PK)      │◀───┐    │ id (PK)          │◀───┐    │ id (PK)          │
│ login_id     │    │    │ slot_date        │    │    │ slot_id (FK)     │
│ password     │    │    │ start_time       │    └────│ member_id (FK)   │
│ name         │    └────│ created_by (FK)  │         │ status           │
│ phone        │         │ end_time         │         │ requested_at     │
│ role         │         │ capacity         │         │ decided_at       │
│ created_at   │         │ is_active        │         │ memo             │
└──────┬───────┘         │ status           │         └────────┬─────────┘
       │                 │ created_at       │                  │
       │                 └──────────────────┘                  │
       │                                                       │
       │            ┌──────────────────────┐                   │
       └───────────▶│    notifications     │◀──────────────────┘
                    ├──────────────────────┤
                    │ id (PK)              │
                    │ recipient_id (FK)    │
                    │ type                 │
                    │ message              │
                    │ is_read              │
                    │ related_reservation_id (FK, nullable) │
                    │ created_at           │
                    └──────────────────────┘
```

**관계 요약**
- `users` 1 : N `time_slots` (사장님이 슬롯을 생성, `created_by`)
- `users` 1 : N `reservations` (회원이 예약을 요청, `member_id`)
- `time_slots` 1 : N `reservations` (한 슬롯에 여러 요청이 올 수 있으나, 확정(CONFIRMED)은 capacity까지만 — 기본 1건)
- `users` 1 : N `notifications` (수신자)
- `reservations` 1 : N `notifications` (관련 예약, nullable)

### 2.2 테이블 명세

#### users — 사용자

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | |
| login_id | VARCHAR(50) | NOT NULL, UNIQUE | 로그인 아이디 |
| password | VARCHAR(255) | NOT NULL | 암호화 저장 (BCrypt) |
| name | VARCHAR(50) | NOT NULL | 이름 |
| phone | VARCHAR(20) | | 연락처 |
| role | VARCHAR(10) | NOT NULL | `ADMIN` / `MEMBER` |
| created_at | DATETIME | NOT NULL | 가입일시 |

#### time_slots — 예약 가능 시간대

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | |
| slot_date | DATE | NOT NULL | 예약 날짜 |
| start_time | TIME | NOT NULL | 시작 시간 |
| end_time | TIME | NOT NULL | 종료 시간 |
| capacity | INT | NOT NULL, DEFAULT 4 | 수용 인원 (가정값 4) |
| is_active | BOOLEAN | NOT NULL, DEFAULT FALSE | 회원 공개 여부 (FR-03) |
| status | VARCHAR(15) | NOT NULL | `AVAILABLE` / `FULL` / `CLOSED` (마감) |
| created_by | BIGINT | FK → users.id | 슬롯 생성자(사장님) |
| created_at | DATETIME | NOT NULL | |

> `status`는 조회 편의를 위한 캐시성 컬럼이다. 마감(`CLOSED`) 판단은 서버에서 `slot_date`와 현재 시각을 비교해 항상 재검증한다 (NFR-04).

#### reservations — 예약

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | |
| slot_id | BIGINT | FK → time_slots.id, NOT NULL | |
| member_id | BIGINT | FK → users.id, NOT NULL | 예약 요청 회원 |
| status | VARCHAR(15) | NOT NULL | `REQUESTED` / `CONFIRMED` / `REJECTED` / `CANCELLED` |
| requested_at | DATETIME | NOT NULL | 요청 일시 |
| decided_at | DATETIME | NULL | 사장님 처리 일시 |
| memo | VARCHAR(255) | NULL | 회원 요청 메모 또는 사장님 거절 사유 |

> 동일 회원이 같은 슬롯에 중복 요청하지 못하도록 `(slot_id, member_id)` 에 대해 `REQUESTED`/`CONFIRMED` 상태인 행이 1개를 넘지 않게 서비스 레이어에서 검증한다.

#### notifications — 알림

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | |
| recipient_id | BIGINT | FK → users.id, NOT NULL | 알림 받는 사람 |
| type | VARCHAR(30) | NOT NULL | `RESERVATION_REQUESTED` / `RESERVATION_CONFIRMED` / `RESERVATION_REJECTED` / `RESERVATION_CANCELLED` |
| message | VARCHAR(255) | NOT NULL | 표시 문구 |
| is_read | BOOLEAN | NOT NULL, DEFAULT FALSE | 읽음 여부 |
| related_reservation_id | BIGINT | FK → reservations.id, NULL | 관련 예약 |
| created_at | DATETIME | NOT NULL | |

### 2.3 상태 값 정리

| 엔터티 | 컬럼 | 값 |
|---|---|---|
| users | role | `ADMIN`, `MEMBER` |
| time_slots | status | `AVAILABLE`(예약가능), `FULL`(정원참), `CLOSED`(마감) |
| reservations | status | `REQUESTED`(요청중), `CONFIRMED`(확정), `REJECTED`(거절), `CANCELLED`(취소) |

## 3. API 설계 (REST)

공통 규칙: 모든 응답은 JSON. 인증 필요한 API는 세션 로그인 상태여야 하며, `ADMIN` 전용은 별도 표기.

### 3.1 인증

| 메서드 | 경로 | 설명 | 권한 |
|---|---|---|---|
| POST | `/api/auth/login` | 로그인 (`loginId`, `password`) | 공개 |
| POST | `/api/auth/logout` | 로그아웃 | 로그인 |
| GET | `/api/auth/me` | 현재 로그인 사용자 정보 | 로그인 |

### 3.2 시간 슬롯 — 사장님 (ADMIN)

| 메서드 | 경로 | 설명 | 대응 FR |
|---|---|---|---|
| POST | `/api/admin/slots` | 슬롯 생성. 단건 또는 시작~종료 시간 + 슬롯 길이로 일괄 생성 | FR-01, FR-02 |
| GET | `/api/admin/slots?date=YYYY-MM-DD` | 특정 날짜의 슬롯 목록 (활성/비활성 모두) | FR-11 |
| PATCH | `/api/admin/slots/{id}` | 슬롯 시간 수정 | FR-02 |
| DELETE | `/api/admin/slots/{id}` | 슬롯 삭제 (예약 존재 시 경고) | UC-01 |
| PATCH | `/api/admin/slots/{id}/activate` | 슬롯 활성화 (회원 공개) | FR-03 |
| PATCH | `/api/admin/slots/{id}/deactivate` | 슬롯 비활성화 | FR-03 |

**POST `/api/admin/slots` 요청 예시**
```json
{
  "date": "2026-05-20",
  "startTime": "10:00",
  "endTime": "18:00",
  "slotMinutes": 120,
  "capacity": 1
}
```
→ 10:00~12:00, 12:00~14:00 ... 형태로 슬롯을 일괄 생성.

### 3.3 시간 슬롯 조회 / 예약 — 회원 (MEMBER)

| 메서드 | 경로 | 설명 | 대응 FR |
|---|---|---|---|
| GET | `/api/slots?date=YYYY-MM-DD` | **활성화된** 슬롯만 조회 (회원용) | FR-04 |
| POST | `/api/reservations` | 예약 요청 (`slotId`, `memo?`) | FR-05 |
| GET | `/api/reservations/me` | 내 예약 목록 + 상태 | FR-10 |
| DELETE | `/api/reservations/{id}` | 예약 취소 (마감 전, 선택 기능) | FR-12 |

**POST `/api/reservations` 처리 로직**
1. 슬롯이 활성화 상태인지 확인.
2. 마감 검증: `slot_date - 1일 23:59:59` 이 지났으면 거절 (FR-09, BR-01).
3. 슬롯 정원(capacity) 내 확정 건수 확인 (BR-02).
4. 동일 회원 중복 요청 확인.
5. `reservations` INSERT (status=`REQUESTED`).
6. 사장님에게 `notifications` INSERT (type=`RESERVATION_REQUESTED`) → FR-06.

### 3.4 예약 처리 — 사장님 (ADMIN)

| 메서드 | 경로 | 설명 | 대응 FR |
|---|---|---|---|
| GET | `/api/admin/reservations?status=REQUESTED` | 처리 대기 예약 목록 | UC-03 |
| PATCH | `/api/admin/reservations/{id}/confirm` | 예약 최종 수락 | FR-07 |
| PATCH | `/api/admin/reservations/{id}/reject` | 예약 거절 (`memo` 사유) | FR-07 |

**confirm / reject 처리 로직**
1. 마감 검증 재확인 (BR-03).
2. `reservations.status` 갱신, `decided_at` 기록.
3. confirm 시: 슬롯 정원이 찼으면 `time_slots.status = FULL`.
4. 회원에게 `notifications` INSERT (`RESERVATION_CONFIRMED` / `RESERVATION_REJECTED`) → FR-08.

### 3.5 알림

| 메서드 | 경로 | 설명 | 대응 FR |
|---|---|---|---|
| GET | `/api/notifications/me` | 내 알림 목록 (안 읽은 것 우선) | FR-06, FR-08 |
| GET | `/api/notifications/me/unread-count` | 안 읽은 알림 개수 (뱃지용) | NFR-03 |
| PATCH | `/api/notifications/{id}/read` | 알림 읽음 처리 | |

## 4. 화면 흐름 (Screen Flow)

### 4.1 사장님(ADMIN) 화면 흐름

```
[로그인]
   │
   ▼
[대시보드] ──── 오늘/이번주 예약 현황 + 미처리 요청 알림 뱃지
   │
   ├──▶ [시간대 관리]
   │       │ 1. 달력에서 날짜 선택
   │       │ 2. 시작/종료/슬롯길이 입력 → 슬롯 일괄 생성
   │       │ 3. 생성된 슬롯 목록에서 [활성화] 토글
   │       └──────────────────────────────────────
   │
   └──▶ [예약 요청 처리]
           │ 1. 알림 클릭 또는 메뉴 진입
           │ 2. 요청 목록(회원/날짜/시간) 확인
           │ 3. [수락] / [거절(사유 입력)] 버튼
           └──────────────────────────────────────
```

### 4.2 회원(MEMBER) 화면 흐름

```
[로그인]
   │
   ▼
[예약 가능 시간대]  ──── 달력에서 날짜 선택 → 활성화된 슬롯 목록
   │
   ▼
[슬롯 선택] ──── 시간 확인, 메모 입력(선택)
   │
   ▼
[예약 요청 완료] ──── "사장님 확인 후 알려드려요" 안내
   │
   ▼
[내 예약 현황] ──── 요청중 / 확정 / 거절 상태 표시 (색상·아이콘 구분)
```

### 4.3 화면 설계 원칙 (NFR-01~03 반영)

- 한 화면에는 **하나의 핵심 행동**만 둔다 (예: "날짜 선택" 화면, "시간 선택" 화면을 분리).
- 버튼은 크고, 텍스트는 명확한 한국어로 ("활성화" 대신 "회원에게 공개하기" 같은 표현 검토).
- 예약 상태는 색상으로 구분: 요청중(노랑) / 확정(초록) / 거절(회색).
- 사장님 대시보드 진입 시 미처리 요청이 가장 먼저, 가장 크게 보이도록 한다.

## 6. 핵심 로직 설계 메모

### 6.1 예약 마감 판정 (FR-09 / BR-01)

```
마감여부(slot) = 현재시각 > (slot.slot_date - 1일) 23:59:59
```
- 회원의 예약 요청(POST /api/reservations), 사장님의 confirm 시점에 **각각 서버에서 검증**한다.
- 배치(스케줄러)로 매일 자정 지난 슬롯의 `status`를 `CLOSED`로 갱신하면 조회 성능에 유리하다 (선택 사항).

### 6.2 동시성 (NFR-04)

- 같은 슬롯에 두 회원이 동시에 요청 → 둘 다 `REQUESTED`까지는 허용. 사장님이 한 명을 `CONFIRMED`하면 정원 초과분은 confirm 단계에서 막는다.
- confirm 트랜잭션에서 슬롯의 확정 건수를 다시 조회해 capacity 초과를 검증한다.

### 6.3 알림 생성 시점

| 이벤트 | 알림 수신자 | type |
|---|---|---|
| 회원이 예약 요청 | 사장님 | `RESERVATION_REQUESTED` |
| 사장님이 수락 | 회원 | `RESERVATION_CONFIRMED` |
| 사장님이 거절 | 회원 | `RESERVATION_REJECTED` |
| 예약 취소(선택) | 상대방 | `RESERVATION_CANCELLED` |

## 7. 미결정 / 추후 논의 사항

| 항목 | 내용 |
|---|---|
| 회원 가입 방식 | 회원이 직접 가입할지, 사장님이 회원을 등록할지 결정 필요 |
| 화면 기술 | Thymeleaf(서버 렌더링) vs HTML+JS(fetch). 학습 목적·앱 확장 고려해 선택 |
| 마감 배치 | 자정 스케줄러로 `CLOSED` 일괄 갱신을 둘지 여부 |
| 외부 알림 | 문자/카카오 알림톡 연동 (확장) |
