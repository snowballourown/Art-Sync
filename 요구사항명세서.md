# 화방 예약 관리 시스템 — 요구사항 명세서 (SRS)

> 프로젝트명: **Art-Sync** (가칭)
> 문서 버전: v1.0
> 작성일: 2026-05-15
> 문서 상태: 초안 (Draft)

---

## 1. 개요

### 1.1 목적

이 문서는 화방(미술 재료 판매·수업 공간)을 운영하는 사장님이 손님(회원)의 방문·수업 시간대를 관리하기 위한 예약 시스템의 요구사항을 정의한다. 본 명세서는 이후 설계·구현·테스트 단계의 기준 문서가 된다.

### 1.2 배경 및 동기

현재 사장님은 회원들의 예약 시간대를 종이 다이어리에 직접 손으로 기록하여 관리하고 있다. 이 방식은 다음과 같은 한계가 있다.

- 사장님 본인만 일정을 알 수 있어 회원이 매번 전화·방문으로 문의해야 한다.
- 중복 예약, 누락, 수기 오류가 발생하기 쉽다.
- 가능한 시간대를 회원에게 미리 보여줄 수 없다.

이를 디지털 예약 시스템으로 대체하여 사장님의 관리 부담을 줄이고, 회원이 스스로 가능한 시간대를 확인·요청할 수 있게 하는 것이 목표다.

### 1.3 프로젝트 범위

| 구분 | 포함 (In Scope) | 제외 (Out of Scope) |
|---|---|---|
| 핵심 | 사장님의 예약 가능 시간대 등록·관리, 회원의 예약 요청, 사장님의 최종 수락/거절, 알림 | 결제, 정산, 매출 통계 |
| 플랫폼 | 웹 (1차 목표) | 모바일 앱 (시간 여유 시 2차 도전) |
| 사용자 | 사장님(관리자) 1명, 회원(손님) 다수 | 직원·다중 관리자 권한 |

### 1.4 용어 정의

| 용어 | 정의 |
|---|---|
| 사장님 / 관리자 | 화방을 운영하며 예약을 최종 승인하는 사용자 |
| 회원 / 손님 | 예약을 요청하는 사용자 |
| 시간 슬롯(Slot) | 예약 단위가 되는 특정 날짜 + 시간 구간 (예: 5월 20일 14:00~16:00) |
| 활성화(Open) | 사장님이 특정 슬롯을 회원에게 예약 가능하도록 공개하는 행위 |
| 예약 요청(Request) | 회원이 활성화된 슬롯에 대해 예약을 신청한 상태 |
| 최종 수락(Confirm) | 사장님이 예약 요청을 검토하여 확정하는 행위 |
| 예약 마감 | 예약을 더 이상 신청·확정할 수 없게 되는 시점. 예약일 기준 **1일 전**. |

### 1.5 참고 자원

- 개발 도구: Claude (구독 사용)
- 기술 스택(예정): Java, Spring, MySQL 또는 PostgreSQL
- 산출물 형태: 웹 애플리케이션 (앱은 추후 확장)

## 2. 전체 설명

### 2.1 제품 관점

Art-Sync는 단일 화방을 위한 독립형 웹 애플리케이션이다. 사장님용 관리 화면과 회원용 예약 화면을 제공하며, 데이터는 하나의 데이터베이스에서 관리된다.

```
[회원 브라우저] ──┐
                  ├──> [웹 서버 / Spring] ──> [DB: MySQL/PostgreSQL]
[사장님 브라우저] ─┘            │
                                └──> [알림 모듈]
```

### 2.2 주요 기능 요약

1. 사장님이 예약 가능한 날짜·시간대를 직접 등록하고 활성화한다.
2. 회원은 활성화된 시간대를 보고 원하는 슬롯에 예약을 요청한다.
3. 예약 요청이 들어오면 사장님에게 알림이 전달된다.
4. 사장님이 요청을 최종 수락하거나 거절한다.
5. 결과가 회원에게 통보된다.
6. 예약 마감(예약일 1일 전)이 지난 슬롯은 더 이상 요청할 수 없다.

### 2.3 사용자 특성

| 항목 | 내용 |
|---|---|
| 주 사용자 연령 | 40~50대 |
| 디지털 숙련도 | 중간~낮음. 복잡한 UI에 익숙하지 않음 |
| 사용 환경 | 주로 스마트폰·PC 브라우저 |
| UI 요구 | **최소한의 터치**로 목표 달성, **직관적이고 이해하기 쉬운 디자인** |

이 특성은 비기능 요구사항(4장)의 사용성 항목에 직접 반영된다.

### 2.4 제약 사항

- 1차 산출물은 웹으로 한정한다.
- 개발자는 1인(학습용 프로젝트)이며, 검증된 기술 스택(Java/Spring/관계형 DB)을 사용한다.
- 관리자는 1명으로 가정한다(다중 관리자 미지원).

### 2.5 가정 및 의존성

- 사장님과 회원 모두 인터넷에 연결된 브라우저를 사용할 수 있다.
- 알림은 1차적으로 시스템 내 알림(웹 화면 내 표시)으로 구현하며, 이메일·문자 등 외부 채널은 확장 사항으로 둔다.
- 회원은 사전에 가입(또는 사장님 등록)된 사용자다.

## 3. 기능 요구사항

각 요구사항은 우선순위(필수/선택)와 출처(기획서 항목)를 함께 표기한다.

### 3.1 액터 정의

| 액터 | 설명 |
|---|---|
| 사장님(Admin) | 시간대 등록·활성화, 예약 요청 최종 수락/거절 |
| 회원(Member) | 활성화된 시간대 조회, 예약 요청, 결과 확인 |
| 시스템(System) | 알림 발송, 예약 마감 시간 검증 |

### 3.2 기능 요구사항 목록

| ID | 요구사항 | 우선순위 | 출처 |
|---|---|---|---|
| FR-01 | 사장님은 예약 가능한 날짜와 시간대(슬롯)를 직접 생성할 수 있다. | 필수 | 기획서 4 |
| FR-02 | 사장님은 시간 슬롯의 길이·시작/종료 시간을 자유롭게 커스텀할 수 있다. | 필수 | 기획서 4 |
| FR-03 | 사장님은 생성한 슬롯을 활성화(공개)하거나 비활성화(숨김)할 수 있다. | 필수 | 기획서 1 |
| FR-04 | 회원은 활성화된 슬롯 목록을 날짜별로 조회할 수 있다. | 필수 | 기획서 1, 2 |
| FR-05 | 회원은 원하는 슬롯을 선택하여 예약을 요청할 수 있다. | 필수 | 기획서 2 |
| FR-06 | 예약 요청이 발생하면 시스템은 사장님에게 알림을 전달한다. | 필수 | 기획서 3 |
| FR-07 | 사장님은 들어온 예약 요청을 최종 수락 또는 거절할 수 있다. | 필수 | 기획서 3 |
| FR-08 | 수락/거절 결과는 회원에게 통보된다. | 필수 | 기획서 3 |
| FR-09 | 시스템은 예약일 기준 1일 전을 예약 마감 시점으로 적용하여, 마감이 지난 슬롯은 요청·수락을 막는다. | 필수 | 기획서 5 |
| FR-10 | 회원은 자신의 예약 요청 상태(요청중/수락/거절)를 확인할 수 있다. | 필수 | 기획서 2, 3 |
| FR-11 | 사장님은 전체 예약 현황(날짜별)을 한눈에 볼 수 있다. | 선택 | 동기 |
| FR-12 | 회원·사장님은 확정된 예약을 취소할 수 있다(마감 전까지). | 선택 | 확장 |

### 3.3 예약 상태 흐름

```
[슬롯 생성] → [비활성]
                │ (사장님 활성화)
                ▼
            [활성/예약가능] ──(회원 예약 요청)──> [요청중]
                                                    │
                          ┌─────────────────────────┤
              (사장님 수락)│                         │(사장님 거절)
                          ▼                         ▼
                      [확정]                     [거절됨]
                          │
                (마감 전 취소 / 선택사항)
                          ▼
                      [취소됨]
```

### 3.4 주요 유스케이스

#### UC-01. 예약 가능 시간대 등록 (사장님)

- **주 액터:** 사장님
- **사전 조건:** 사장님이 로그인되어 있다.
- **기본 흐름:**
  1. 사장님이 시간대 관리 화면에 진입한다.
  2. 날짜를 선택한다.
  3. 시작 시간, 종료 시간, 슬롯 길이를 입력하여 슬롯을 생성한다(FR-01, FR-02).
  4. 생성된 슬롯을 활성화한다(FR-03).
  5. 시스템이 해당 슬롯을 회원에게 공개한다.
- **대안 흐름:** 이미 예약이 있는 슬롯은 삭제·수정 시 경고를 표시한다.

#### UC-02. 예약 요청 (회원)

- **주 액터:** 회원
- **사전 조건:** 회원이 로그인되어 있고, 활성화된 슬롯이 존재한다.
- **기본 흐름:**
  1. 회원이 날짜별 예약 가능 시간대를 조회한다(FR-04).
  2. 원하는 슬롯을 선택한다.
  3. 예약 요청 버튼을 누른다(FR-05).
  4. 시스템이 마감 시간(예약일 1일 전)을 검증한다(FR-09).
  5. 요청이 등록되고 사장님에게 알림이 전달된다(FR-06).
- **예외 흐름:**
  - 4단계에서 마감이 지난 경우, "예약이 마감되었습니다" 메시지를 표시하고 요청을 막는다.
  - 동일 슬롯에 이미 다른 요청/확정이 있는 경우 정책에 따라 처리한다(3.5 참고).

#### UC-03. 예약 최종 수락/거절 (사장님)

- **주 액터:** 사장님
- **사전 조건:** 처리 대기 중인 예약 요청이 있다.
- **기본 흐름:**
  1. 사장님이 알림을 통해 요청 화면에 진입한다.
  2. 요청 내용(회원, 날짜, 시간)을 확인한다.
  3. 수락 또는 거절을 선택한다(FR-07).
  4. 시스템이 예약 상태를 갱신하고 회원에게 결과를 통보한다(FR-08).

### 3.5 비즈니스 규칙

| ID | 규칙 |
|---|---|
| BR-01 | 예약 마감은 **예약일 0시 기준 24시간 전**(= 예약일 전날)으로 한다. (구현 시 정확한 기준 시각은 설계 단계에서 확정) |
| BR-02 | 하나의 슬롯에 대한 동시 예약 허용 인원은 기본 4명으로 한다. 슬롯 생성 시 사장님이 변경할 수 있다. |
| BR-03 | 마감이 지난 슬롯은 요청·수락·취소가 모두 불가능하다. |
| BR-04 | 비활성 슬롯은 회원 화면에 노출되지 않는다. |

## 4. 비기능 요구사항

| ID | 분류 | 요구사항 |
|---|---|---|
| NFR-01 | 사용성 | 주 사용자가 40~50대임을 고려하여, 핵심 작업(시간대 등록, 예약 요청)은 **최소한의 터치/클릭**으로 완료할 수 있어야 한다. |
| NFR-02 | 사용성 | 글자 크기, 버튼 크기를 충분히 크게 하고, 한 화면에 너무 많은 정보를 담지 않는다. 직관적인 용어를 사용한다. |
| NFR-03 | 사용성 | 예약 상태(요청중/확정/거절)는 색상·아이콘 등으로 한눈에 구분되어야 한다. |
| NFR-04 | 신뢰성 | 중복 예약·마감 검증은 서버에서 반드시 재확인하여 데이터 정합성을 보장한다. |
| NFR-05 | 성능 | 일반적인 조회·요청은 2초 이내에 응답한다. |
| NFR-06 | 보안 | 사장님과 회원의 권한을 분리하고, 로그인한 사용자만 기능을 사용할 수 있다. |
| NFR-07 | 이식성 | 1차는 웹 브라우저(모바일·PC)에서 동작하며, 이후 앱 확장을 고려해 화면·API를 분리 설계한다. |
| NFR-08 | 유지보수성 | Java/Spring 표준 구조를 따르고, 기능 단위로 모듈을 분리한다. |

## 5. 데이터 요구사항 (개념 수준)

상세 ERD는 설계 단계에서 작성하되, 개념 수준의 주요 엔터티는 다음과 같다.

| 엔터티 | 주요 속성 | 설명 |
|---|---|---|
| 사용자(User) | id, 이름, 연락처, 역할(사장님/회원), 로그인 정보 | 시스템 사용자 |
| 시간 슬롯(TimeSlot) | id, 날짜, 시작시간, 종료시간, 활성화 여부, 상태 | 사장님이 만든 예약 단위 |
| 예약(Reservation) | id, 슬롯 id, 회원 id, 상태(요청중/확정/거절/취소), 요청일시 | 회원의 예약 요청 및 결과 |
| 알림(Notification) | id, 수신자 id, 내용, 읽음 여부, 생성일시 | 사장님·회원에게 전달되는 알림 |

## 6. 향후 확장 (Out of Scope, 참고용)

- 모바일 앱(네이티브 또는 하이브리드) 버전
- 이메일·카카오톡·문자 등 외부 알림 채널
- 그룹 수업(다인 예약) 지원
- 매출·방문 통계 대시보드
- 정기 예약(반복 일정) 기능
